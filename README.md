# Техническое задание на разработку программного продукта для планирования встреч

## 1. Общие описание
Необходимо разработать программный продукт для планирования встреч сотрудников внутри компании. Система состоит из клиентского мобильного приложения и серверной части.

## 2. Клиентская часть (Android-приложение)
Клиентом является **Android-приложение**, которое должно быть реализовано на **Kotlin** или **Java** с использованием **адаптивной верстки**.

Приложение должно предоставлять интуитивно понятный интерфейс, в котором реализован следующий функционал:

*   **Регистрация и авторизация** сотрудника.
*   **Настройка личного профиля**.
*   **Создание встречи** с возможностью приглашения других сотрудников. Приглашенные сотрудники должны иметь возможность дать бинарный ответ (**принял** / **не принял**).
*   **Просмотр списка активных приглашений** на встречи с возможностью дать ответ о возможности посещения.
*   **Просмотр собственного расписания встреч** в разрезе дня, недели и месяца.

**Важное условие:** временные слоты для встреч разбиты строго по часам. Например, можно создать встречу на 9:00–10:00, но создание встречи на 9:10, 9:05 и т.д. — невозможно.

## 3. Серверная часть
Сервер представляет собой **микросервис на Spring Boot**, работающий с СУБД **PostgreSQL** или **H2**.

Серверное приложение должно:
*   Предоставлять API, необходимый для полноценной работы мобильного приложения.
*   Осуществлять все основные операции с данными в базе через **Spring Data JPA**.
*   Использовать **Liquibase** для создания схемы базы данных и ее предзаполнения.

# UserStory И UseCase

## 1) Создание учетной записи (регистрация)

**User story:**  
Как *Android приложение* я хочу отправить данные регистрации на сервер, чтобы сервер создал нового пользователя.

**Критерии принятия:**
- Сервер принимает данные регистрации: email/login, пароль, имя, фамилия.
- Пароли хранятся в виде хеша.
- При успешной регистрации сервер возвращает 201 и DTO пользователя.
- При существующем логине возвращается ошибка 409 Conflict.
- При проблемах с подключением к БД — 500 Internal Server Error.

**Use-case: Регистрация пользователя**
- **Основниой участник:** Mobile App
- **Предусловия:** Пользователь не авторизован; сервер доступен; БД работает.
- **Основной поток:**
    1. App -> `POST /api/v1/auth/register` с body `{ "email","password","firstName","lastName" }`.
    2. Сервер валидирует поля, создаёт нового пользователя, хеширует пароль, сохраняет в БД.
    3. Сервер возвращает 201 с DTO пользователя.
- **Альтернативный поток:**
    - Некорректные данные → 400 Bad Request.
    - Логин уже существует → 409 Conflict.
    - Ошибка БД → 500 Internal Server Error.

## 2) Просмотр расписания встреч

**User story:**  
Как *Android приложение* я хочу получить список встреч пользователя за выбранный период, чтобы показать расписание в приложении.

**Критерии принятия:**
- API возвращает встречи, сгруппированные по часам (слоты 9:00, 10:00 и т.д.).
- Поддерживаются фильтры: день, неделя, месяц.
- Если встреч нет — возвращается пустой массив.
- Ошибки сервера возвращают 5xx, ошибки клиента — 4xx.

**Use-case: Получение расписания**
- **Основной участник:** Mobile App
- **Условия:** Пользователь авторизован; сервер и БД доступны.
- **Основной поток:**
    1. App -> `GET /api/v1/calendar?userId={id}&from={start}&to={end}`
    2. Сервер выбирает встречи пользователя (организатор или приглашённый, статус ≠ DECLINED), агрегирует по дате и часу.
    3. Сервер возвращает 200 с массивом встреч.
- **Альтернативный поток:**
    - Нет встреч → 200 с пустым массивом.
    - Ошибка сервера/БД → 500 Internal Server Error.

## 3) Создание встречи и приглашение сотрудников

**User story:**
Как *Android приложение* я хочу отправить данные новой встречи на сервер, чтобы сервер создал встречу и уведомил приглашённых сотрудников.

**Критерии принятия:**
- Сервер принимает дату, время начала, тему, участников.
- Встреча создаётся только на полный час (9:00, 10:00 и т.д.).
- Проверяется конфликт времени для организатора (и при необходимости для обязательных участников).
- При успешном создании возвращается 201 с DTO встречи и приглашений.

**Use-case: Создание встречи**
- **Основной участник:** Mobile App
- **Предусловия:** Пользователь авторизован; сервер и БД доступны.
- **Основной поток:**
    1. App -> `POST /api/v1/meetings` с body:
       ```json
       {
         "title":"...",
         "date":"2026-02-02",
         "startHour":9,
         "durationHours":1,
         "organizerId":123,
         "invitees":[{"userId":124,"required":true},{"userId":125,"required":false}]
       }
       ```
    2. Сервер проверяет правильность часа, конфликты.
    3. Сервер создаёт запись `meeting` и записи `invitation` для участников, коммитит транзакцию.
    4. Асинхронно уведомляет участников, возвращает 201 с DTO встречи и приглашений.
- **Альтернативный поток:**
    - Неполный час → 400 Bad Request.
    - Конфликт с другой встречей → 409 Conflict с деталями.

## 4) Просмотр входящих приглашений

**User story:**  
Как *Android приложение* я хочу получить список активных приглашений пользователя, чтобы показать их в приложении.

**Критерии принятия:**
- API возвращает список активных приглашений с основной информацией о встрече.
- Если приглашений нет — возвращается пустой массив.
- Ошибки сервера/сети обрабатываются с кодами 4xx/5xx.

**Use-case: Получение активных приглашений**
- **Основной участник:** Mobile App
- **Предусловия:** Пользователь авторизован; сервер доступен.
- **Основной поток:**
    1. App -> `GET /api/v1/invitations?userId={id}&status=PENDING`
    2. Сервер выбирает все активные приглашения пользователя.
    3. Возвращает 200 с массивом invitation DTO.
- **Альтернативный поток:**
    - Нет приглашений → 200 с пустым массивом.
    - Ошибка сервера/сети → 500 Internal Server Error или 503 Service Unavailable.

## 5) Ответ на приглашение (ACCEPT / DECLINE)

**User story:**  
Как *Android приложение* я хочу отправить на сервер ответ на приглашение, чтобы сервер сохранил решение и обновил расписание пользователя.

**Критерии принятия:**
- Можно принять или отклонить приглашение.
- При принятии проверяется отсутствие конфликта по времени.
- После успешного ответа обновляется статус в БД и уведомляется организатор.

**Use-case: Ответ на приглашение**
- **Основной участник:** Mobile App
- **Предусловия:** Пользователь авторизован; приглашение существует и активно.
- **Основной поток:**
    1. App -> `POST /api/v1/meetings/{meetingId}/invitations/{invitationId}/response` `{ "status":"ACCEPTED" }`
    2. Сервер проверяет права, проверяет конфликты, обновляет `invitation.status` и `responded_at`.
    3. Возвращает 200 с updated invitation DTO; уведомляет организатора.
- **Альтернативный поток:**
    - Ошибка сети → 503 Service Unavailable.
    - Встреча отменена → 410 Gone.
