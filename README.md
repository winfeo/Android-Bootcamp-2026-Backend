# Техническое задание на разработку программного продукта для планирования встреч

## Команда: Название Съели

## 1. Общие описание
Необходимо разработать программный продукт для планирования встреч сотрудников внутри компании. Система состоит из клиентского мобильного приложения и серверной части.

## 2. Клиентская часть (Android-приложение)
Клиентом является **Android-приложение**, которое должно быть реализовано на **Kotlin** или **Java** с использованием **адаптивной верстки**.

Приложение должно предоставлять интуитивно понятный интерфейс, в котором реализован следующий функционал:

*   **Регистрация и авторизация** сотрудника.
*   **Настройка личного профиля**.
*   **Создание встречи** с возможностью приглашения других сотрудников. Приглашенные сотрудники должны иметь возможность дать бинарный ответ (**принял** / **не принял**).
*   **Просмотр списка активных приглашений** на встречи с возможностью дать ответ о возможности посещения.
*   **Просмотр собственного расписания встреч** в разрезе дня, недели и месяца.

**Важное условие:** временные слоты для встреч разбиты строго по часам. Например, можно создать встречу на 9:00–10:00, но создание встречи на 9:10, 9:05 и т.д. — невозможно.

## 3. Серверная часть
Сервер представляет собой **микросервис на Spring Boot**, работающий с СУБД **PostgreSQL** или **H2**.

Серверное приложение должно:
*   Предоставлять API, необходимый для полноценной работы мобильного приложения.
*   Осуществлять все основные операции с данными в базе через **Spring Data JPA**.
*   Использовать **Liquibase** для создания схемы базы данных и ее предзаполнения.

# Декомпозиция задачи
## 1. User Stories
* Как сотрудник, я хочу зарегистрироваться, чтобы создать учётную запись для использования приложения.
* Как сотрудник, я хочу авторизоваться, чтобы получить доступ к своему профилю и функциям приложения.
* Как сотрудник, я хочу редактировать данные в своём профиле, чтобы информация обо мне была актуальной.
* Как сотрудник, я хочу создавать новые встречи с другими сотрудниками, чтобы обсуждать важные задачи в назначенное время.
* Как сотрудник, я хочу видеть список активных приглашений на встречи с возможностью принять или отклонить каждое, чтобы предоставить инициатору встречи информацию о своём присутствии или отсутствии.
* Как сотрудник, я хочу просматривать своё расписание встреч в календаре с режимами отображения по дням, неделям или месяцам, чтобы иметь представление о запланированных встречах. 
 
## 2. Use Cases
## Use Case: Регистрация нового сотрудника
**Цель:** Создать новую учётную запись сотрудника.  
**Акторы:** Сотрудник.  
**Предусловия:** Сотрудник не имеет учетной записи в системе.    
**Основной поток:**
1. Сотрудник переходит с экрана входа на экран регистрации.
2. Отображается форма с полями для заполнения: `ФИО`, `email`, `пароль`.
3. Сотрудник заполняет форму данными.
4. Приложение отправляет данные на сервер.
5. Сервер проверят корректность введённых данных и возвращает токен авторизации.
6. Приложение сохраняет токен и переходит на главный экран.

**Альтернативный поток:**
1.	Пользователь вводит `email`, который уже используется.
2.	Сервер возвращает ошибку регистрации.
3.	Приложение отображает ошибку регистрации.

## Use Case: Авторизация сотрудника
**Цель:** Авторизация сотрудника для доступа к функционалу приложения.  
**Акторы:** Сотрудник.  
**Предусловия:** Учётная запись сотрудника уже создана в системе.  
**Основной поток:**  
1. Приложение отображает форму для ввода: `email` и `пароль`.
2. Пользователь вводит учётные данные.
3. Сервер проверят учётные данные и возвращает токен авторизации.
4. Приложение сохраняет токен и переходит на главный экран.

**Альтернативный поток:**
1. Пользователь вводит неверные учётные данные.
2. Сервер возвращает ошибку.
3. Приложение отображает сообщение о неверных данных.


## Use Case: Редактирование профиля
**Цель:** Актуализировать личную информацию.  
**Акторы:** Сотрудник.  
**Предусловия:** Сотрудник авторизован в приложении.  
**Основной поток:**
1. Приложение отображает текущие данные на странице профиля.
2. Пользователь нажимает на кнопку редактирования информации.
3. Приложение отображает форму для изменения данных.
4. Пользователь изменяет поля в форме.
5. Пользователь подтверждает изменение данных.
6. Сервер проверяет корректность новых данных и возвращает статус проверки.
7. Приложение отображает сообщение об успешном изменении данных.
8. Пользователь переходит на страницу профиля с обновлёнными данными.
   
**Альтернативный поток 1:**
1. Пользователь вводит неверные данные, либо уже используемые данные другими пользователями.
2. Сервер возвращает соответствующую ошибку.
3. Приложение отображает сообщение об ошибке.

**Альтернативный поток 2:**
1. Пользователь подтверждает выход из формы редактирования профиля через диалоговое окно.
2. Данные не отправляются на сервер, данные не изменяются.
3. Пользователь переходит на страницу профиля.

## Use Case: Создание новой встречи
**Цель:** Запланировать встречу на конкретный временной слот.  
**Акторы:** Сотрудник.  
**Предусловия:** Сотрудник авторизован в приложении.  
**Основной поток:**
1. Приложение отображает календарь встреч пользователя.
2. Пользователь нажимает на кнопку создания новой встречи.
3. Сервер возвращает доступные даты, часовые слоты и пользователей для добавления.
4. Приложение отображает форму для создания встречи с полями для заполнения.
5. Сервер сохраняет данные о встрече.
6. Сервер отправляет уведомления участникам, указанным при создании встречи.
7. Приложение отображает сообщение об успешном создании встречи.
   
**Альтернативный поток:**
1. Пользователь отменят создание встречи.
2. Данные не отправляются на сервер, новая встреча не создаётся.

## Use Case: Просмотр и ответ на приглашения
**Цель:** Подтвердить или отклонить приглашение на встречу.  
**Акторы:** Сотрудник.  
**Предусловия:** Сотрудник авторизован в приложении.  
**Основной поток:**
1. Пользователь открывает страницу с входящими приглашениями.
2. Приложение отображает список приглашений.
3. Пользователь нажимает на приглашение для детального просмотра.
4. Пользователь нажимает «Принять».
5. Данные пользователя: `user_id` и его положительное решение в участии отправляются на сервер.
6. Сервер обновляет статус участия пользователя в встрече.
7. Сервер добавляет встречу в календарь приглашенного пользователя.
8. Приложение отображает уведомление о добавлении новой встречи в календарь.

**Альтернативный поток 1:**
1. Пользователь нажимает «Не принять».
2. Данные пользователя: `user_id` и его негативное решение в участии отправляются на сервер.
3. Сервер обновляет статус участия пользователя в встрече.

**Альтернативный поток 2:**  
Страница пустая, новых приглашений для просмотра нет.


## Use Case: Просмотр расписания встреч
**Цель:** Просмотреть список запланированных встреч на выбранный период.   
**Акторы:** Сотрудник.  
**Предусловия:** Сотрудник авторизован в приложении.  
**Основной поток:**
1. Сотрудник открывает страницу с календарём.
2. Приложение отображает все встречи сотрудника (по умолчанию режим отображения - `день`)
3. Система отображает все встречи сотрудника за выбранный период (`день`, `неделя` или `месяц`).
4. Пользователь нажимает на конкретную встречу в расписании.
5. Приложение открывает страницу с подробной информацией о встрече.

**Альтернативный поток 1:**  
Приложение не отображает встречи сотрудника (пользователь не добавлен ни в одну встречу ещё). 

## 3. Задачи backend-разработчика
1. Реализовать API для регистрации и авторизации:
    * Эндпоинт `POST /api/auth/register` для создания новой учетной записи сотрудника.
    * Эндпоинт `POST /api/auth/login` для проверки учетных данных.
    * `JWT` / `Basic authentication` для управления авторизацией.
2. Реализовать API для работы с сотрудниками:
    * Эндпоинт `GET /api/employees` для получения списка сотрудников.
    * Эндпоинт `GET /api/employees/{user_id}` для получения данных текущего авторизованного сотрудника.
    * Эндпоинт `PUT /api/employees/{user_id}` для обновления данных профиля.
3. Реализовать API для работы со встречами:
    * Эндпоинт `GET /api/meetings` для получения списка всех созданных встреч.
    * Эндпоинт `POST /api/meetings` для создания новой встречи.
    * Эндпоинт `GET /api/meetings/{user_id}` для получения расписания пользователя с поддержкой параметра `period` (day / week / month).
    * Эндпоинт `GET /api/meetings/{meeting_id}` для получения детальной информации о конкретной встрече.
4. Реализовать API для работы с приглашениями:
    * Эндпоинт `GET /api/invitations/{user_id}` для получения списка активных приглашений для пользователя.
    * Эндпоинт `PUT /api/invitations/{user_id}` для отправки ответа на приглашение (принять / не принять). 
6. Написать миграции базы данных с Liquibase (таблицы сотрудников, учётных записей, встреч, статуса участия, временных слотов, дней).
7. Настроить Spring Security для разграничения доступа.
